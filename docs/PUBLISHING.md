# Публикация и доставка обновлений

Документ описывает надежный порядок публикации расширения через GitHub и защиту авторских прав.

## Модель распространения

Рекомендуемая модель: source-available, personal use only.

Это значит:

- пользователи могут скачать и поставить официальную версию;
- код можно видеть в репозитории для доверия и проверки;
- проект не является open-source;
- сторонние сборки, перепубликация, продажа, удаление автора и клоны запрещены.

Такой режим лучше подходит для проекта, где важно дать пользователям доступ к расширению, но не отдавать право свободно копировать и переиздавать работу.

## Что публиковать в GitHub

Публиковать:

- `manifest.json`
- `src/`
- `assets/`
- `data/`
- `scripts/`
- `README.md`
- `LICENSE.md`
- `NOTICE.md`
- `docs/`

Не публиковать:

- `tmp/`
- архивы `.zip`, `.rar`, `.7z`
- собранные `.crx`
- приватные ключи `.pem`
- локальные папки браузера и редактора

Эти исключения уже добавлены в `.gitignore`.

## Первый релиз

1. Создать репозиторий на GitHub.
2. Назвать, например: `bk-interactive-map`.
3. В описание репозитория добавить:

   `Интерактивная карта подземелий БК: синхронизация персонажа, маршруты, объекты, NPC и дроп. Personal use only.`

4. Загрузить файлы проекта.
5. Проверить, что в репозиторий не попали `tmp/`, архивы и приватные ключи.
6. Создать тег версии, например `v0.1.1`.
7. Создать GitHub Release с описанием из `docs/RELEASE_TEMPLATE.md`.
8. В скролле дать ссылку на официальный релиз и инструкцию установки.

## Обновления

Каждый релиз должен иметь:

- новую версию в `manifest.json`;
- тег GitHub вида `v0.1.2`, `v0.1.3` и так далее;
- короткое описание изменений;
- список исправленных ошибок;
- пометку, нужно ли пользователям полностью перезагружать вкладку игры.

Минимальный порядок:

1. Внести изменения.
2. Проверить синтаксис:

   ```powershell
   node --check src\content.js
   node --check src\page-bridge.js
   node --check src\background.js
   node --check src\dungeons.js
   ```

3. Обновить версию в `manifest.json`.
4. Создать commit.
5. Создать tag.
6. Опубликовать GitHub Release.
7. Написать в скролле короткую новость об обновлении.

## Как пользователям обновляться

Пока расширение распространяется как распакованное:

1. Скачать новую версию из GitHub Releases.
2. Распаковать в папку расширения.
3. Открыть `chrome://extensions`.
4. Нажать `Обновить`.
5. Перезагрузить вкладку игры.

## Автоматические обновления в будущем

Для полностью автоматических обновлений есть два нормальных пути:

1. Chrome Web Store.

   Самый удобный вариант для пользователей. Браузер сам обновляет расширение, но нужна публикация в магазине и прохождение проверки.

2. Self-hosted CRX update.

   Можно распространять подписанный `.crx` и `updates.xml` на своем хостинге. Это сложнее: нужен приватный ключ, стабильный URL обновлений и аккуратная подпись версий.

Для старта через GitHub лучше использовать ручные GitHub Releases. Это проще, прозрачнее и безопаснее для первого публичного запуска.

## Защита авторских прав

Для защиты уже добавлены:

- `LICENSE.md` с запретом перепубликации, продажи, удаления автора и сторонних сборок;
- `NOTICE.md` с обязательным уведомлением об авторе;
- копирайт-уведомления в исходных файлах;
- `.gitignore`, чтобы случайно не публиковать архивы, временные файлы и приватные ключи.

Дополнительно перед публикацией рекомендуется:

- закрепить в `README.md`, что проект не является open-source;
- оставить ссылки на автора и поддержку в интерфейсе расширения;
- публиковать релизы только из официального репозитория;
- просить пользователей скачивать только официальные версии;
- при копировании проекта отправлять жалобу в GitHub через механизм copyright/DMCA.

## Важно

Публичный JavaScript в браузерном расширении нельзя технически защитить на 100 процентов: пользователь может открыть файлы расширения после установки. Поэтому надежная защита строится не на скрытии кода, а на авторском праве, лицензии, публичной атрибуции, официальном канале релизов и быстрой реакции на копии.
